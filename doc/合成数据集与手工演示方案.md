# 合成数据集与手工演示方案

## 1. 结论

当前项目可以做手工演示走查，即使暂时没有真实数据集也可以推进。

原因有三点：

1. 当前前后端主链路已经具备 Candidate 与 HR 的基本闭环。
2. AI Service 当前支持 `RESUME_PARSER_PROVIDER=mock`，可以在没有真实模型和真实简历样本的情况下完成解析状态推进。
3. 浏览器端 Wasm PDF 预处理链路已经接通，因此答辩时仍然可以展示“浏览器端先处理，再上传”的核心亮点，而不需要依赖真实候选人数据。

因此，现阶段最合理的做法不是等待真实数据，而是尽快建立一套“明确标注为 Synthetic / LLM-generated”的演示数据集与实验样本集。

---

## 2. 数据策略

建议把数据分成三层，不要只准备一种：

### 2.1 演示 PDF 简历

用途：

- 走 Candidate 上传主链路。
- 展示浏览器端 Wasm PDF 预处理。
- 展示上传后进入解析队列、解析完成、生成推荐。

特点：

- 文件形态必须是 PDF。
- 内容必须像真实简历，但完全虚构。
- 每份简历最好控制在 1 到 2 页，便于答辩演示。

### 2.2 结构化解析真值表

用途：

- 做“解析字段正确性”对照。
- 做实验表格，而不是只截图界面。

特点：

- 每份简历都要有一份人工确认后的标准答案。
- 标准答案字段建议和当前系统最常展示的字段一致。

建议至少包含：

- 姓名
- 邮箱
- 电话
- 技能列表
- 教育经历
- 工作经历
- 摘要

### 2.3 岗位样本集

用途：

- 走 HR 创建岗位与推荐评估链路。
- 做 Candidate 反向匹配演示。
- 做 Top-3 或 Top-5 合理性评价。

特点：

- 岗位数量不需要太多，但要覆盖高匹配、中匹配、低匹配三类结果。

---

## 3. 最小可用样本规模

如果目标是先完成答辩级演示，不建议一开始做很大。

推荐最小规模：

- 8 份合成简历 PDF
- 8 份对应真值 JSON
- 6 个岗位

样本分布建议：

- 后端开发方向：2 份
- 前端开发方向：2 份
- 测试 / 运维 / 数据方向：2 份
- 明显跨方向样本：2 份

岗位分布建议：

- Kotlin / Java 后端岗位：2 个
- Vue / 前端岗位：2 个
- 数据 / 测试 / 综合岗位：2 个

这样可以稳定造出三种效果：

1. 高匹配：推荐理由明显成立。
2. 中匹配：有优势也有技能缺口，适合展示 XAI 建议。
3. 低匹配：适合展示“为什么不匹配”。

---

## 4. LLM 生成约束

如果后续用 LLM 生成简历数据，必须加上约束，避免生成无效样本。

建议约束如下：

1. 不允许出现真实公司、真实学校、真实手机号、真实邮箱。
2. 姓名统一使用中文化虚构名，例如“林泽远”“周清禾”。
3. 邮箱统一用保留域名，例如 `example.com`。
4. 电话统一使用虚构规则号段，例如 `1380000xxxx`。
5. 工作经历和项目经历要围绕本系统已有岗位方向生成，不要发散到医学、法律等无关领域。
6. 简历内容长度要可控，尽量保证 1 到 2 页 PDF 可读。
7. 每份简历都要明确标签：目标岗位、强项、短板、预计匹配岗位。

---

## 5. 推荐的 LLM 生成产物

每个候选人样本建议一次生成 3 份内容：

1. 简历 Markdown 版
2. 解析真值 JSON 版
3. 样本说明卡片

其中：

- Markdown 用于后续导出 PDF
- JSON 用于实验对照
- 样本说明卡片用于答辩时快速介绍这个样本为什么被这样推荐

---

## 6. 简历生成提示词模板

下面这个提示词适合直接发给 LLM 生成简历内容。

```text
你要生成一份完全虚构、不可追溯到任何真实个人的中文技术简历。

要求：
1. 输出为 Markdown。
2. 候选人姓名、邮箱、电话、学校、公司都必须是虚构信息，不得使用真实人物或真实机构。
3. 邮箱必须使用 example.com 域名。
4. 电话使用 1380000xxxx 形式。
5. 简历目标岗位为：{目标岗位}。
6. 技术方向为：{技术方向}。
7. 工作年限为：{工作年限}。
8. 必须包含：基本信息、个人摘要、技能清单、工作经历、项目经历、教育经历。
9. 内容长度控制在 1 到 2 页简历可承载的范围。
10. 风格要像真实校招 / 社招简历，但不能出现任何真实隐私。

同时满足以下设计目标：
- 强项：{强项}
- 短板：{短板}
- 预期最匹配岗位：{预期匹配岗位}
- 预期次匹配岗位：{预期次匹配岗位}
```

## 7. 真值 JSON 生成提示词模板

下面这个提示词适合让 LLM 根据已经生成的简历 Markdown 再输出一份人工校对用真值结构。

```text
请根据下面这份虚构简历，提取结构化真值数据。

输出要求：
1. 只输出 JSON。
2. 字段包括：
   - candidateName
   - email
   - phone
   - summary
   - skills
   - workExperiences
   - educationExperiences
3. `skills` 必须是字符串数组。
4. 工作经历和教育经历必须保留时间、组织、角色或专业。
5. 不要补充简历中不存在的信息。

简历内容如下：
{简历 Markdown 内容}
```

## 8. 手工演示走查建议流程

即使没有真实数据，也可以完成一套完整手工走查。

### 8.1 Candidate 主路径

1. 注册一个 Candidate 账号。
2. 进入个人资料页，补 GitHub 和作品集链接。
3. 上传一份合成 PDF 简历。
4. 展示浏览器端 Wasm PDF 预处理结果：页预览、文本摘要、轻量引用。
5. 提交上传，观察简历状态从 `PENDING_PARSE` 进入解析。
6. 解析完成后进入推荐页。
7. 打开岗位详情，查看匹配度、优势、风险、建议。
8. 执行投递、收藏或忽略。
9. 查看时间线和投递记录。

### 8.2 HR 主路径

1. 登录 HR 账号。
2. 创建或打开岗位。
3. 查看推荐结果。
4. 打开岗位申请审核台。
5. 查看候选人最新简历摘要和匹配解释。
6. 把一个候选人状态改为约面或淘汰，并填写备注。
7. 回到 Candidate 侧验证状态同步。

## 9. 当前最适合的演示模式

如果你现在就要录演示或答辩彩排，建议采用下面模式：

1. 前端走真实页面。
2. 后端走真实接口。
3. Redis 保持开启。
4. AI Service 使用 `mock` 解析器。
5. 简历文件使用 LLM 生成的虚构 PDF。

这种模式的优点：

- 主链路是真实的，不是假按钮。
- 数据是虚构的，不涉及隐私和授权问题。
- 风险最低，不依赖老师提供真实样本。
- 论文里也能合理说明“实验与演示数据为合成样本集”。

---

## 10. 论文与答辩表述建议

如果没有真实数据集，论文和答辩里不要回避，应该明确说明：

1. 本系统演示与实验使用的是合成样本集。
2. 合成样本由人工设计岗位方向，再由 LLM 生成候选人简历初稿。
3. 所有样本均经过人工检查与脱敏约束，不包含真实个人隐私。
4. 合成样本主要用于验证系统链路可用性、解析字段覆盖度、推荐合理性与演示稳定性。
5. 该策略是对“缺少真实授权数据集”这一现实条件的工程化替代方案。

这个表述是合理的，关键是你后面要把样本设计规则、真值表、评估表做扎实。

---

## 11. 下一步建议

最优先的不是先等数据，而是立刻做下面三件事：

1. 先确定 6 个岗位模板。
2. 再生成 8 份虚构简历 Markdown 和对应真值 JSON。
3. 最后导出其中 3 到 5 份 PDF，专门用于 Candidate 演示链路。

只要这三步完成，手工演示走查就可以正式开始。
