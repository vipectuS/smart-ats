# Smart ATS 总控推进计划

本文档作为本项目后续推进的唯一主计划文件，目标不是“写一个大概路线”，而是把后面所有需要做的事情尽量收拢到一个地方，方便严格按清单推进、验收和收尾。

适用场景：
- 日常开发排期。
- 阶段验收。
- 答辩前查漏补缺。
- 后续任何 Agent 或开发者接手时的统一执行依据。

---

## 0. 项目目标与当前判断

### 0.1 最终目标

本项目最终要交付的不是单个模块，而是一套可完整演示的智能招聘辅助系统，至少要满足下面四件事：

1. 有完整业务闭环。
   - 候选人能注册、上传简历、查看推荐、投递岗位。
   - HR 能创建岗位、查看简历、发起解析、查看匹配和解释结果。
2. 有多模态亮点。
   - 简历上传必须体现浏览器端 Wasm 预处理能力。
   - AI 服务必须能把多源输入统一成结构化 TalentProfile。
3. 有可解释性。
   - 推荐结果不仅要有分数，还要能解释来源、风险和建议。
4. 有论文证据。
   - 需要测试、实验、截图、演示路径，而不是只停留在“工程能跑”。

### 0.2 当前状态判断

当前项目不是从零开始，而是已经从“主业务链路能演示”推进到“答辩级最小交付可复现”的状态。

按仓库中的真实实现、测试和材料看，当前更接近：

- 已达到答辩级最小交付，但仍不是工程完备态。

当前已真实落地的部分：

- 仓库结构清理和文档入口收口。
- 后端主业务链路、候选人动作链路、推荐与审核主干。
- AI 服务的解析、嵌入、结构化 Job Fit Report 接口。
- 前端 Candidate 与 HR 两条主路径页面及主要联调入口。
- 浏览器端 Wasm PDF 预处理主链路、Phase 3 截图与运行采集脚本。
- 合成数据集、真值 JSON、PDF/HTML/GitHub 演示素材首包。
- 管理员独立最小交付：系统概览、技能词典维护、解析失败记录、Admin Console 页面。
- 统一启动、演示导入与证据导出脚本，可在本地生成 JSON 证据结果。
- 本地配置与公开配置拆分。

当前最影响最终验收结论的缺口：

- Phase 6 已补出最小证据链，但论文化表格、人工标注说明和更完整基线对比仍可继续增强。
- Phase 7 已补出统一启动与演示数据入口，但仍未形成完整版答辩话术、录屏编排和容器化部署方案。
- 多模态链路在本地演示中仍以 synthetic truth-aware mock 为主，当前更适合证明可复现闭环，不等同于真实商业多模态效果评测。

当前已核验的运行证据：

- 前端生产构建通过。
- AI 服务自动化测试通过。
- 后端自动化测试通过，少量依赖 Docker 的向量集成测试在当前环境中跳过。
- `tools/maintenance/bootstrap_local_demo.sh --with-demo-data` 已在当前环境实跑成功，并生成演示导入摘要与 Phase 6 JSON 报告。

### 0.3 当前最高优先级

1. 继续把 Phase 6 的 JSON 证据结果整理成论文可引用表格、截图和文字说明。
2. 收口 Phase 7 的 3 分钟 / 8 分钟答辩演示脚本与录屏顺序。
3. 如时间允许，再补容器化部署或更正式的基线对比实验。

## 1. 强约束与执行原则

### 1.1 强约束

1. 前端“上传简历”必须体现浏览器端 Wasm 架构能力，这是项目亮点之一。
2. 浏览器端必须先把 PDF 渲染为图像或轻量化中间结果，再上传给后端或 AI 服务。
3. 不允许把沉重原始 PDF 直接推送给后端作为正式主流程。
4. 即使为了短期调试保留临时降级方案，也必须把 Wasm 主路径作为最终方案写入实现、文档和答辩材料。
5. 前端联调必须遵守接口契约优先原则，不能猜接口路径、响应结构或 DTO 字段。
6. 每完成一个明显阶段，必须同步更新对应模块 CHANGELOG 和必要文档。

### 1.2 执行原则

1. 一次只收口一个阶段，不多线发散。
2. 每个阶段都必须有可演示结果，而不是只提交代码。
3. 每个阶段都必须有最小验证方式：编译、测试、接口验证或页面走查。
4. 任何新增功能都必须能服务于演示、论文或答辩，不做纯炫技功能。
5. 如果某项工作无法在时间内做完整，就明确降级为扩展项，不做半成品。

## 2. 总体执行顺序

后续严格按照下面顺序推进：

1. Phase 1：联调收口与接口统一。
2. Phase 2：前端完整用户旅程收尾。
3. Phase 3：多模态最小闭环，重点是 Wasm PDF 主路径。
4. Phase 4：匹配与 XAI 强化。
5. Phase 5：管理员最小版。
6. Phase 6：测试、实验、论文支撑。
7. Phase 7：部署与答辩演示打磨。

阻塞关系：

- Phase 1 不完成，Phase 2 和 Phase 3 的联调成本会持续放大。
- Phase 3 不完成，多模态亮点和毕业设计核心卖点无法成立。
- Phase 4 不完成，推荐系统“可解释”部分说服力不足。
- Phase 6 不完成，工程即使能跑也不足以支撑论文答辩。

## 7. Phase 3：多模态最小闭环

### 7.1 目标

满足毕业设计“多模态”要求的最小可信版本，重点不是覆盖所有输入，而是先把一个亮点链路做扎实。

### 7.2 范围定义

正式纳入本阶段的输入源：

1. PDF 简历。
2. HTML 作品集链接。

当前 Phase 3 起点已经具备：Candidate 上传页可在浏览器端完成 PDF 逐页渲染、缩略图预览、文本摘要抽取与轻量引用生成，因此后续重点不再是“有没有渲染”，而是“如何把浏览器端结果稳定送入多模态解析主链路”。
3. GitHub 仓库链接。

### 7.2.1 当前进展补充

- 已补通候选人资料中的 `githubUrl` / `portfolioUrl` 到 AI 解析消息的透传链路：候选人重新触发解析时，后端现在会把这些外链作为 `externalContentReferences` 一并发送给 AI 服务。
- AI 服务现在会对上述外链执行受控的 HTML 文本提取，并把提取结果与浏览器端 PDF 预处理文本、分页预览图片、后端简历引用一起送入统一解析提示。
- GitHub 路径已从“页面纯文本”升级到“最小结构化字段集”：当前 AI 服务会优先整理仓库 owner/repo、描述、README 摘要、语言和近期公开信号，再把这些字段并入统一解析提示；下一步不再是补字段本身，而是决定这些字段是否同步进入嵌入链路与后续匹配评分。

可选扩展项：

1. DOCX 简历。
2. 更多作品集站点适配。
3. 更复杂的 GitHub 贡献度分析。
