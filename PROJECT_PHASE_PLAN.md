# Smart ATS 接手与推进阶段计划

本文档面向后续接手该项目的开发者，目标是把当前“后端较成熟、AI 服务中等成熟、前端联调未完全收口”的状态，推进到一个可答辩、可演示、可量化评估的毕业设计成品。

## 0. 当前判断

当前项目的真实状态不是“从零开始”，而是“主链路已成型，但前端收尾、多模态落地、实验评估不足”。

当前优先级判断：
1. 先收口联调，而不是继续扩功能。
2. 先补多模态最小闭环，而不是追求大而全。
3. 先补论文所需量化评估，再考虑高级优化。

## 1. 阶段总览

| 阶段 | 目标 | 产出 | 预计优先级 |
| --- | --- | --- | --- |
| Phase 1 | 结构收口与联调修复 | 干净目录、统一 API 适配、可跑通主流程 | 最高 |
| Phase 2 | 前端功能收尾 | 完整 HR / Candidate 演示路径 | 最高 |
| Phase 3 | 多模态解析最小闭环 | PDF / HTML / GitHub 链接最小可用解析 | 最高 |
| Phase 4 | 匹配与 XAI 强化 | 更稳定的评分与解释输出 | 高 |
| Phase 5 | 系统管理最小版 | 管理员基础能力、配置与日志视图 | 中 |
| Phase 6 | 测试、实验与论文支撑 | 定量评估、基线对比、答辩材料 | 最高 |
| Phase 7 | 部署与演示打磨 | 一键启动、演示数据、截图录屏 | 高 |

## Phase 1. 结构收口与联调修复

### 目标
- 让仓库结构可读、职责清晰、入口稳定。
- 消除“页面已写但接口没接好”的不确定状态。
- 停止继续堆叠遗留脚本和临时产物。

### 任务
1. 统一前端 API 调用方式。
   - 所有页面统一走 `frontend/src/utils/api.ts`。
   - 禁止同一项目同时混用 axios 解包和手写 fetch 裸读响应包。
   - 统一后端响应读取方式：`{ status, data, message }`。
2. 修复前端已知断点。
   - 接通注册页真实接口。
   - 修复简历列表页分页数据读取错误。
   - 接通候选人投递记录页真实接口。
   - 把未纳入路由的页面纳入正式导航或删除。
3. 统一文档入口。
   - 根目录 README 只保留项目总览、运行入口、目录说明。
   - 子模块 README / GUIDE 各自负责本模块开发说明。
4. 清理生成物与历史脚本。
   - 保持 `tools/maintenance/` 仅用于少量可复用维护工具，不保留一次性补丁脚本。
   - 避免再次把一次性修复脚本放到项目根目录。

### 验收标准
- 前端能正常登录、进入 HR 看板、查看岗位详情、触发评估。
- 候选人端能查看推荐、投递、收藏、忽略、查看时间线。
- README 与实际目录一致，没有明显失效说明。

## Phase 2. 前端功能收尾

### 目标
- 把前端从“能构建”提升到“能稳定演示”。

### 任务
1. 补全导航与角色路由。
   - HR 与 Candidate 分开布局或至少分开导航入口。
   - 登录后按角色跳转到正确首页。
2. 完成 Candidate 用户旅程。
   - 注册。
   - 个人资料编辑。
   - 上传简历。
   - 查看岗位推荐。
   - 投递/撤回/收藏/取消收藏/忽略/取消忽略。
   - 查看投递记录与时间线。
3. 完成 HR 用户旅程。
   - 登录。
   - 看板查看。
   - 岗位列表。
   - 岗位详情。
   - 调权重重新评估。
   - 简历列表与解析状态轮询。
4. UI 收口。
   - 去掉默认 mock 文案。
   - 去掉硬编码 Admin User 头像和名称。
   - 补统一空态、错误态、加载态。

### 验收标准
- 两种角色都有一条完整的点击路径，不需要手工改 token 或改 URL。
- 页面上不再出现“等待后端开放”“模拟成功”这类答辩风险文案。

## Phase 3. 多模态解析最小闭环

### 目标
- 满足毕业设计“多模态”要求的最小可信版本，而不是追求全量工业级方案。

### 建议实现范围
1. 文本型 PDF 简历。
   - 前端可先上传文件引用，后端/AI 服务读取文本。
   - 如果 Wasm PDF 预处理来不及，可先交付服务端抽取版，再在论文中说明 Wasm 为下一步优化方向。
2. HTML 作品集链接。
   - 复用 AI 服务现有 `content_extractor.py` 的 HTML 文本提取。
3. GitHub 链接。
   - 最小版只抓公开仓库 README、语言占比、近期提交统计或项目描述。
   - 不要一上来做复杂代码贡献度分析，否则时间会失控。
4. DOCX 支持。
   - 如果时间足够，补 `python-docx` 或等价方案。
   - 如果时间不够，明确标注为扩展项，不要半完成。

### 关键任务
1. 定义输入类型和来源。
   - 简历文件。
   - 作品集 URL。
   - GitHub URL。
2. 在 AI 服务中补一个统一内容聚合层。
   - 原始文本。
   - 外链内容。
   - 结构化元信息。
3. 把聚合结果送入统一解析提示词，输出严格 TalentProfile JSON。
4. 为至少 3 组真实样例保留输入与输出示例，供论文和答辩展示。

### 验收标准
- 至少可稳定演示 1 份 PDF 简历、1 个作品集页面、1 个 GitHub 链接进入统一画像。
- TalentProfile 输出字段稳定，不靠人工补 JSON。

## Phase 4. 匹配与 XAI 强化

### 目标
- 把当前已经可用的推荐逻辑打磨成“可解释、可展示、可复现”。

### 任务
1. 固化评分公式。
   - 规则分。
   - 语义分。
   - 权重归一化。
   - 评分区间说明。
2. 固化解释结构。
   - 匹配优势。
   - 风险点。
   - 缺失技能。
   - 提升建议。
3. 前端图形化展示结构化分解。
   - 分项得分条形图或标签卡片。
   - 与雷达图保持一致。
4. 对推荐结果进行样例复核。
   - 随机抽 10 组岗位-候选人配对，人工检查排序是否合理。

### 验收标准
- 任一推荐结果都能回答“为什么是这个分数”。
- HR 调权重后，结果变化能被界面和文案同时解释。

## Phase 5. 系统管理最小版

### 目标
- 满足任务书中的管理员模块要求，做到“有而且说得过去”。

### 建议范围
1. 管理员基础数据维护。
   - 技能标签库。
   - 学校/学历字典库，二选一也可。
2. 角色与权限展示。
   - 至少能说明 HR / Candidate / Admin 的差异。
3. 运行日志最小展示。
   - 解析任务日志。
   - 失败回调日志。
   - 系统操作日志任选其一。

### 验收标准
- 有一个真实的管理员页面或 API，不只是数据库里存在 ADMIN 角色。

## Phase 6. 测试、实验与论文支撑

### 目标
- 把工程完成度转化为毕业设计可交付材料。

### 必做项
1. 数据集与样本说明。
   - 公开简历样本来源。
   - 模拟岗位样本来源。
   - 样本规模。
2. 解析准确率评估。
   - 至少选若干关键字段：姓名、技能、教育、工作经历。
   - 给出人工标注对照。
3. 匹配合理性评估。
   - Top-3 或 Top-5 人工评价。
   - 与关键词匹配基线对比。
4. 性能评估。
   - 解析平均耗时。
   - 推荐接口耗时。
   - 看板接口耗时。
5. 故障与边界说明。
   - LLM 超时。
   - JSON 格式错误。
   - 外链无法访问。

### 产出建议
- 在 `doc/` 下补一份实验评估文档。
- 形成可直接复用到论文中的表格和截图。

### 验收标准
- 能用 2 到 4 页材料完整回答“系统效果如何证明”。

## Phase 7. 部署与演示打磨

### 目标
- 让系统具备稳定答辩演示能力。

### 任务
1. 提供一键启动方式。
   - `docker-compose` 或启动脚本，二选一。
2. 准备演示数据。
   - 默认 HR 用户。
   - 默认 Candidate 用户。
   - 3 到 5 个岗位。
   - 3 到 5 份已解析简历。
3. 准备答辩脚本。
   - 3 分钟版本。
   - 8 分钟版本。
4. 录制短演示视频或准备截图序列。

### 验收标准
- 新环境下可在有限步骤内完成演示。
- 不依赖临场手改数据库。

## 2. 我会怎么接手这个项目

如果由我继续接手，我会按下面顺序推进，而不是同时多线发散：

1. 第一周只做联调收口。
   - 修注册、简历页、投递记录页、角色路由。
   - 删除或隐藏所有 mock 文案。
2. 第二周只做多模态最小闭环。
   - 先把 PDF 文本和 HTML 作品集打通。
   - GitHub 只做最小公共信息提取。
3. 第三周做实验评估。
   - 构造样本、跑指标、写实验文档。
4. 第四周再做管理员最小版和部署打磨。

核心原则：
- 不再继续增加“看起来高级但无法证明价值”的功能。
- 所有新增功能都必须服务于答辩演示或论文证据。
- 优先做“能展示、能解释、能量化”的东西。

## 3. 可直接执行的下一步

建议按下面顺序继续：
1. 修复前端联调断点。
2. 实现多模态最小闭环。
3. 补实验评估文档与脚本。
4. 最后再做部署和答辩打磨。
